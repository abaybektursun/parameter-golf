# Kernel Speedrun Learnings

- 2026-03-20: The previously committed `experiments/004_kernel_speedrun/results.tsv` and `learnings.md` belonged to an older 9-layer kernel-speedrun script. The new staged 11-layer / 3x-MLP baseline was committed as `d506433` before continuing experiments.
- 2026-03-20: Commit `d506433` (`kernel speedrun: import 11-layer baseline`) reached `step:84/150 val_loss:5.6079 val_bpb:3.3213 train_time:120669ms step_avg:1436.53ms` on 1xA100 with `ITERATIONS=150`, `VAL_LOSS_EVERY=0`, `MAX_WALLCLOCK_SECONDS=120`.
- 2026-03-20: This script's nominal speed benchmark path is slow operationally because, after the 120-second training cap is reached, it still performs a full validation pass before printing the final `val_loss/step_avg` line, and then falls through into serialization and roundtrip eval. For iteration, it is sufficient to watch the in-training `step_avg` lines and stop after a stable reading near the cap.
- 2026-03-20: Commit `d9eb73d` (`kernel speedrun: restore inductor gemm autotune`) re-enabled `torch._inductor.config.max_autotune=True` and `max_autotune_gemm=True`. Startup/autotune time increased materially on 1xA100, but steady-state training improved to `1423.03ms/step` by step 80 (vs. `1436.53ms` baseline), an improvement of about `13.50ms` / `0.94%`. Kept.
- 2026-03-20: A direct port attempt of modded-nanogpt's Triton `FusedLinearReLUSquareFunction` into a standalone compile smoke test failed under `torch.compile(fullgraph=True)`. The upstream implementation relies on `triton.tools.tensor_descriptor.TensorDescriptor.from_tensor(...)`, and Dynamo rejected tracing its `data_ptr() % 16` alignment check. An eventual port likely needs to be wrapped as a custom op or otherwise hidden from Dynamo.
- 2026-03-20: A micro `TORCH_COMPILE_DEBUG=1` compile of the plain `relu(fc(x)).square()` MLP showed that Inductor already fuses the forward `relu + square` elementwise chain into one Triton pointwise kernel after the first GEMM. Backward still materializes the hidden activation path and uses separate GEMMs, so the remaining upside is likely in backward traffic / buffer elimination rather than forward epilogue fusion.
- 2026-03-20: Commit `213c031` (`kernel speedrun: keep castedlinear weights in bf16`) removed the setup-time `module.float()` pass over `CastedLinear` modules, keeping those weights in bf16 while leaving `restore_low_dim_params_to_fp32` intact. Steady-state speed improved again to `1412.20ms/step` by step 80 on 1xA100 (vs. `1423.03ms` at `d9eb73d`), an additional improvement of about `10.83ms` / `0.76%`. Kept.
- 2026-03-20: Commit `04a1904` (`kernel speedrun: skip no-op castedlinear casts`) added dtype checks in `CastedLinear.forward` so `weight.to(x.dtype)` / `bias.to(x.dtype)` are skipped when the tensors already match the activation dtype. With the bf16-weight change already in place, this reduced steady-state speed further to `1408.08ms/step` by step 80 on 1xA100 (vs. `1412.20ms` at `213c031`), another gain of about `4.12ms` / `0.29%`. Kept.
- 2026-03-20: Commit `a45f3d9` (`kernel speedrun: keep control tensors in bf16`) removed the `restore_low_dim_params_to_fp32(...)` calls so control tensors such as `q_gain`, residual scales, skip weights, and the smear gate remain in bf16. Startup got a bit slower, but steady-state speed still improved to `1401.86ms/step` by step 80 on 1xA100 (vs. `1408.08ms` at `04a1904`), another gain of about `6.22ms` / `0.44%`. Kept.
- 2026-03-20: Commit `9b598f5` (`kernel speedrun: enable max-autotune compile mode`) was operationally unattractive on 1xA100. The benchmark never emitted `warmup_step:1/20` after more than four minutes, so even before comparing steady-state speed it was a discard for this loop. Reverted in `71c17d7`.
- 2026-03-20: Commit `4aea0fa` (`kernel speedrun: recompute relu squared mlp backward`) replaced the plain `relu(fc(x)).square()` MLP autograd path with a custom compiled autograd function that recomputes the hidden activation during backward instead of saving the full 1536-wide buffer. The custom path matched the naive eager implementation on local CUDA smoke tests and improved steady-state speed to `1379.35ms/step` by step 80 on 1xA100 (vs. `1401.86ms` at `a45f3d9`), a gain of about `22.51ms` / `1.61%`. Kept.
- 2026-03-20: Commit `809d559` (`kernel speedrun: recompute softcapped ce backward`) wrapped the tied-embedding softcapped cross-entropy in the same style of custom autograd recomputation so logits are rebuilt in backward instead of saved from forward. CUDA smoke tests showed numerically close behavior and `torch.compile(fullgraph=True)` compatibility, but steady-state speed regressed to `1384.75ms/step` by step 80 on 1xA100 (vs. `1379.35ms` at `4aea0fa`), about `5.40ms` / `0.39%` slower. Reverted in `841c158`.
- 2026-03-20: A focused `TORCH_COMPILE_DEBUG=1` compile of `CausalSelfAttention.forward` on the current branch showed that Inductor already fuses each Q/K projection's RMSNorm + RoPE + elementwise postprocessing into Triton kernels (`triton_per_fused__fused_rms_norm__unsafe_view_add_arange_cat_cos_mul_neg_sin_slice_unsqueeze_view_*`). The debug trace still showed the manual GQA path lowering to `unsqueeze -> expand -> clone` before FlashAttention.
- 2026-03-20: Commit `0dcc95d` (`kernel speedrun: use sdpa native gqa`) switched the attention path to `F.scaled_dot_product_attention(..., enable_gqa=True)` to avoid the explicit `repeat_interleave` on K/V heads. CUDA smoke tests showed exact agreement with the old path and fullgraph compile support, but steady-state speed came in at `1379.69ms/step` by step 80 on 1xA100 (vs. `1379.35ms` at `4aea0fa`), effectively a wash and about `0.34ms` / `0.02%` slower. Reverted in `af442ea`.
- 2026-03-20: Commit `59118ec` (`kernel speedrun: add muon single-rank fast path`) bypassed Muon's flattened update buffer and all-reduce staging when `world_size == 1`, which removed about 52 MB of per-step staging traffic on this local benchmark and matched the reference 1-rank update math in a CUDA smoke test. The full speed run still landed at `step:87/150 val_loss:5.5869 val_bpb:3.3089 train_time:120170ms step_avg:1381.26ms`, which is `1.91ms` / `0.14%` slower than the `4aea0fa` best, so it was reverted in `fc140fb`.
- 2026-03-20: Commit `c478c6c` (`kernel speedrun: try reduce-overhead compile mode`) crashed before `warmup_step:1/20`. Inductor's CUDA-graphs path raised `RuntimeError: Error: accessing tensor output of CUDAGraphs that has been overwritten by a subsequent run`, pointing at the cached RoPE `freqs.cos()` / `freqs.sin()` tensors in `Rotary.forward`. The error message explicitly suggests either cloning those tensors outside `torch.compile()` or calling `torch.compiler.cudagraph_mark_step_begin()` before each compiled model invocation. Reverted in `b52e061`.
- 2026-03-20: Commit `54b1819` (`kernel speedrun: mark cudagraph steps for reduce-overhead`) added `torch.compiler.cudagraph_mark_step_begin()` before warmup, training, validation, and compiled-logits calls while keeping `mode="reduce-overhead"`. It still crashed before `warmup_step:1/20` with the same overwritten-output error on the RoPE cache tensors, so the step markers were not sufficient in this code path. Reverted in `41bda8d`.
- 2026-03-20: Commit `cee35c1` (`kernel speedrun: clone rope cache for reduce-overhead`) kept `mode="reduce-overhead"` but returned cloned RoPE cache outputs to try to break the aliasing that Inductor's CUDA-graphs path complained about. It still crashed before `warmup_step:1/20` with the same overwritten-output error pointing at `self._cos_cached = freqs.cos()[None, :, None, :]`, so cloning the returned tensors was not enough to satisfy cudagraph lifetime tracking. Reverted in `4873cb9`.
- 2026-03-20: Commit `74a4a14` (`kernel speedrun: add triton softcapped ce kernel`) replaced the softcap + cross-entropy portion of the LM-head loss with a custom Triton row-wise kernel wrapped in a custom autograd function, while keeping the BF16 `F.linear` matmul for logits and the BF16 GEMMs for `grad_x` / `grad_weight`. Small CUDA smoke tests showed close loss/gradient agreement with the baseline implementation and a tiny `torch.compile(fullgraph=True)` toy module worked, but end-to-end training regressed massively: after warmup it was already at `step:10/150 train_time:21532ms step_avg:2153.21ms`, about `773.86ms` / `56.10%` slower than the `4aea0fa` best. Reverted in `fd61720`.
- 2026-03-20: An uncommitted local experiment on top of `ce6ec10` ported modded-nanogpt's Triton `linear_relu_square` kernel behind `torch.library.custom_op` wrappers so Dynamo no longer traced `TensorDescriptor.from_tensor(...)`, while keeping the current recompute strategy in `ReLUSquaredMLPFunction.backward`. Small CUDA smoke tests showed very small mean output/gradient deltas versus the BF16 baseline and a toy `torch.compile(fullgraph=True)` module now compiled successfully, so the wrapper approach did fix the earlier traceability problem. End-to-end training still regressed badly: startup took about two minutes before `warmup_step:1/20`, and after warmup the run settled at `step:10/150 train_time:33800ms step_avg:3380.02ms`, about `2000.67ms` / `145.05%` slower than the `4aea0fa` best. The likely issue is that this tensor-descriptor/TMA-flavored kernel is tuned for the modded-nanogpt/H100 stack and is a poor fit for our 1xA100 benchmark, so further MLP work should avoid this exact `TensorDescriptor` custom-op port.
- 2026-03-20: The best current speed benchmark for the new 11-layer kernel-speedrun model is still `1379.35ms/step` on 1xA100 at commit `4aea0fa` (the current branch head `ce6ec10` keeps the same code by reverting only the slower follow-up experiments).
- Next kernel-focused candidates for this new baseline:
- A true fused linear + softcapped cross-entropy kernel or custom op. The pure PyTorch backward-recompute version still loses to the MLP-only path, so the remaining upside likely requires eliminating more of the logits-path memory traffic than recomputation alone does.
- If MLP work is revisited, avoid the upstream `TensorDescriptor`/custom-op path on this 1xA100 setup. Any future MLP kernel attempt should use a different implementation strategy than the modded-nanogpt descriptor kernel.
- Attention-side work should move past Q/K RMSNorm + RoPE, since Inductor is already fusing that path. Any future attention experiment should target a hotspot not already covered by the current generated Triton kernels.
- `torch.compile(..., mode="reduce-overhead")` looks like a dead end on this baseline. Plain mode, explicit step markers, and cloned RoPE outputs all still crashed before warmup on the same CUDA-graphs lifetime issue, so further time should go back to non-cudagraph kernel work.
